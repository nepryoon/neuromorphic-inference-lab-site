This memo captures the engineering decisions behind the demo: retrieval traceability, evaluation signals, and reliability patterns that distinguish production-ready RAG from a toy chatbot.

What this system demonstrates

Retrieval-first design, transparent traces, and evaluation-minded iterations. The demo is static for a public portfolio, but mirrors a production flow: ingestion → chunking → embeddings → vector index → retrieval → answer generation → monitoring.

Evaluation signals (LLMOps)

A credible RAG build ships with measurable signals: retrieval hit rate, precision@k, citation correctness, faithfulness, latency, and cost per query. The portfolio narrative should include failure cases and the iteration that fixed them.

precision@k faithfulness latency budget cost controls

Reliability patterns

Guardrails that matter in real deployments: prompt injection defence, source-based claims only, refusal on empty retrieval, PII redaction, structured output validation, and monitoring for drift/regressions. CI/CD should cover retriever changes, prompt changes, and evaluation set updates.
