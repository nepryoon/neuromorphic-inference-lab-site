Design choices

This system prioritizes demonstrable operational skills: data engineering patterns (PostgreSQL + JSONB), training reproducibility (Docker + versioned artifacts), serving discipline (FastAPI + health checks), and observability (Prometheus + Grafana). Each component is production-minded.

Technical skills demonstrated

Data Engineering

  • API Integration: Real-time ingestion from CityBikes and Open-Meteo public APIs
  • Raw Storage: PostgreSQL with JSONB for flexible schema evolution
  • Schema Management: SQL initialization scripts for repeatable setup
  • Feature Pipeline: Pandas-based transformation logic separated from training

ML Development

  • Training Pipeline: scikit-learn with reproducible Docker-based builds
  • Artifact Management: Versioned model.joblib + metadata.json + metrics.json
  • Quality Thresholds: Adaptive quantile-based targets to handle class imbalance
  • Synthetic Testing: DB-free training tests for CI reliability

Operationalization

  • Inference API: FastAPI with OpenAPI documentation
  • Health Endpoints: Liveness checks + model status reporting
  • DB-backed Predictions: Latest feature snapshot for automated inference
  • Manual Features: Explicit feature input for testing/debugging

Monitoring & Observability

  • Prometheus Metrics: Request counts, latency distributions (P95/P99), model version
  • Grafana Dashboards: Visual monitoring of service health and performance
  • Alert Rules: ModelNotLoaded, HighErrorRate, HighPredictionLatency, InferenceServiceDown
  • AlertManager: Configured for webhook notifications (Slack/PagerDuty ready)

Repo structure & workflow

The repository is organized for clarity and operational discipline.

Directory layout

services/
  ├── ingestion/     # Data collection from CityBikes + Open-Meteo
  ├── trainer/       # Model training with artifact output
  └── inference/     # FastAPI serving with Prometheus metrics

db/
  └── init.sql       # Schema initialization

monitoring/
  ├── prometheus/    # Prometheus config + alert rules
  └── grafana/       # Grafana provisioning

tests/
  ├── test_inference_contract.py    # API schema verification
  └── test_trainer_synthetic.py     # DB-free training test

.github/workflows/
  ├── ci.yml                        # CI pipeline with quality gate
  └── scheduled-retrain.yml         # Daily retraining workflow

docker-compose.*.yml   # Orchestration files for each service layer

Local development workflow

  1. Start stack: docker compose -f docker-compose.local.yml -f docker-compose.monitoring.yml -f docker-compose.data.yml up
  2. Run ingestion: docker compose run --rm ingestion
  3. Train model: docker compose -f docker-compose.train.yml run --rm trainer
  4. Query API: curl http://localhost:8000/health
  5. View metrics: http://localhost:9090 (Prometheus)
  6. View dashboards: http://localhost:3000 (Grafana)

CI/CD workflow

  1. Push to repository triggers CI pipeline
  2. pytest runs contract + synthetic training tests
  3. Quality gate asserts F1 >= 0.60
  4. Artifacts uploaded for inspection
  5. Scheduled workflow handles daily retraining
  6. S3 upload + ECS deployment automated

Risks & mitigations

Identified risks

  • Public API availability: CityBikes/Open-Meteo may experience downtime
  • Data drift: Bike availability patterns change over time
  • Model staleness: Without retraining, performance degrades
  • Deployment complexity: Multi-container orchestration requires infrastructure

Mitigations

  • Graceful fallback: Inference API returns informative errors on model load failure
  • Scheduled retraining: Daily workflow refreshes model with latest data
  • Quality gates: CI prevents deployment of poorly-performing models
  • Monitoring alerts: Prometheus rules catch service degradation early
